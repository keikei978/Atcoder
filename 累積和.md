# 累積和（Cumulative Sum / Prefix Sum）

> ひとことで言うと: 「区間の合計」を何回も聞かれるとき、前計算した表の**引き算1回**で即答するテク。`O(N×Q)` → `O(N+Q)`。

---

## いつ使うか（見分け方）

次の3点セットが揃ったら累積和を疑う。

1. 「区間 L〜R の合計／個数」を聞かれる
2. それが**クエリで何回も**飛んでくる（Q がデカい）
3. 素朴に毎回足すと **TLE する**

「合計 × 何回も × そのままだと間に合わない」= 累積和。

---

## 基本アイデア

数列があるとする。

```
番号:  1   2   3   4   5
値:    3   1   4   1   5
```

「先頭から i 番目までの合計」を並べた表を作る。これが累積和 `S`。

```
S[0] = 0                    ← 番兵（何も足してない）
S[1] = 3            = 3
S[2] = 3+1          = 4
S[3] = 3+1+4        = 8
S[4] = 3+1+4+1      = 9
S[5] = 3+1+4+1+5    = 14
```

作り方は「ひとつ前の S に、今の値を足すだけ」。

```
S[i] = S[i-1] + 値[i]
```

`S[0] = 0` を必ず置くのがコツ（後の引き算で楽になる）。

---

## 区間の合計の出し方

「L〜R の合計」が欲しいとき。例えば 2〜4番（1+4+1=6）。

```
S[4] = 3+1+4+1     ← 1〜4 の合計
S[1] = 3           ← 1 番だけの合計
S[4] - S[1] = 9 - 3 = 6   ✓
```

つまり：

```
L〜R の合計 = S[R] - S[L-1]
```

S[R] は 1〜R の合計、S[L-1] は 1〜(L-1) の合計。引けば L〜R だけが残る。**引き算1回**で出る。

### なぜ S[0]=0 が必要か
L=1 のとき S[L-1] = S[0] になる。ここが 0 じゃないと計算が壊れる。だから番兵として 0 を置く。

---

## なぜ速いのか（愚直との比較）

ここが累積和の核心。「あらかじめ足しておく」だけでなぜ爆速になるか。

```
愚直解：毎回足し直す
┌─────────────────────────────┐
│ クエリ1：2〜6 を全部足す      │
│ クエリ2：1〜5 を全部足す      │
│ クエリ3：3〜7 を全部足す      │
│ …毎回ゼロから足し直し         │
│ 1クエリで最悪 N 回            │
└─────────────────────────────┘
   合計：N × Q ≈ 100億回 → TLE

────────────────────────────────

累積和：前計算して引き算
┌─────────────────────────────┐
│ 前計算（1回だけ）             │
│   S[i] = S[i-1] + 値 を N回   │
│   → 表が完成                  │
├─────────────────────────────┤
│ 各クエリ（引き算1回）          │
│   クエリ1：S[6] - S[1]        │
│   クエリ2：S[5] - S[0]        │
│   …使い回すだけ。O(1)         │
└─────────────────────────────┘
   合計：N + Q ≈ 20万回 → AC
```

| | 愚直 | 累積和 |
|---|---|---|
| 計算量 | O(N × Q) | O(N + Q) |
| 最悪の操作回数 | 約100億 | 約20万 |
| 2秒制限 | ❌ TLE | ✅ 余裕 |

**掛け算 (N×Q) が 足し算 (N+Q) に変わる**のが効いている。Q が大きいほど差が爆発的に開く（この例で約5万倍）。

愚直は同じような区間の足し算を毎回ゼロからやり直している。累積和は「**前計算でまとめて1回払い、あとは引き算で使い回す**」。重複計算を消すのが本質。

---

## DPではない（混同注意）

累積和は `S[i] = S[i-1] + a[i]` という漸化式の形を持つので、一見 DP に見える。

| | 累積和 | DP |
|---|---|---|
| 式 | `S[i] = S[i-1] + a[i]` | `dp[i] = max(dp[i-1], dp[i-2]+a[i])` など |
| 性質 | ただ足し上げる**一本道** | 複数の選択肢から**最適を選ぶ** |
| 判断 | なし | max/min・場合分けあり |

累積和には「選ぶ」要素がない。だから普通は DP とは呼ばず、**前計算テクニックの一種**と捉える。漸化式という形が共通しているだけ。

## 実装テンプレ（Python）

```python
# 数列 a（1-indexed で扱う）の累積和
N = int(input())
a = list(map(int, input().split()))   # 長さ N

S = [0] * (N + 1)        # S[0]=0 の番兵込み
for i in range(1, N + 1):
    S[i] = S[i-1] + a[i-1]   # a は0-indexedなので a[i-1]

# 区間 [L, R]（1-indexed）の合計
def range_sum(L, R):
    return S[R] - S[L-1]
```

## 応用パターン

| パターン | 内容 |
|---|---|
| 区間和クエリ | 「L〜R の合計は？」を何回も。基本形 |
| 条件を満たす区間の個数 | 「和がちょうど K の連続部分列はいくつ？」累積和＋ハッシュ |
| 二次元累積和 | グリッドで長方形領域の合計を高速に出す |
| いもす法 | 「区間に一律 +1」を何回もやってから最終結果を見る（差分の累積和） |
